2 · VPS FastAPI bridge

Single-tenant, single-process FastAPI. Holds exactly one active MT5 session at a time — every POST /connect replaces the previous one. This is why the Next.js server serialises VPS access with an in-memory mutex.

GET /healthLiveness + MT5 connected flagOptional X-API-Key
POST /connectLog into an MT5 account (login, investor password, server)Tears down previous session first
GET /accountBalance, equity, leverage, currencyRequires connected session
GET /tradesOpen positions + recent closed deals (lookback hours)Only OUT deals returned (entry==1)
GET /historyFull IN+OUT deals between optional from/to datesDefault from 2000-01-01

FA->>MT: history_deals_get(naive UTC) MT->>M5: deals[] M5-->>MT: IN + OUT deals MT-->>FA: normalized list FA-->>NX: {deals[], meta}
⚑ Single-session VPS. Any concurrent /connect from two users would thrash the MT5 session. The mutex protects correctness but means sync throughput is capped at one account at a time across the whole deployment.

Optimisation hotspots in the sync pipeline

  • Full history every run. vpsHistory fetches from 2000-01-01 by default. Switch to incremental pull using accounts.last_synced_at so payloads shrink as history grows.
  • In-memory mutex. lib/sync-mutex.ts is a single-process promise queue; multi-worker deployments (Vercel functions scale horizontally) will bypass it. Replace with a Supabase advisory lock or Redis lock.
  • Sessions rebuild. Full re-aggregation on every sync is O(trades). Do an incremental UPSERT keyed on dates touched by new trades, or move to a Postgres view / materialised view.
  • Sequential cron. /api/cron/sync-all loops accounts serially because of the single-session VPS. Either shard across multiple VPSes per broker or queue jobs with backoff.
  • Batch size = 500. Works, but a 6 MB Supabase payload limit means very wide rows (journal text, screenshot URLs arrays) can overflow. Consider dynamic sizing.
  • Partial closes. normalizer keeps only the last OUT deal. Aggregate lot-weighted close price instead so reported pnl_pips is accurate.

Key constraints and RLS

  • Unique keys prevent duplicate syncs: trades(account_id, position_id), sessions(account_id, date), mt5_credentials(user_id, login).
  • Manual trades use a dedicated sequence manual_trade_position_seq producing negative position_ids so they never collide with MT5 ticket IDs during upsert.
  • RLS: service_role has full access (used by API routes). User-facing policies scope by auth.uid() = user_id, and trades/sessions join through accounts.
  • Cascades: deleting an account cascades to trades and sessions. Deleting an auth user cascades to all user-scoped tables.
⚑ RLS EXISTS joins. The trades and sessions policies use EXISTS (SELECT 1 FROM accounts ...). Make sure accounts(user_id) and trades(account_id) are indexed (they are) — but also consider denormalising user_id onto trades so RLS can short-circuit without a join when query volume grows.

  • Rate limit is soft: it returns 429 on /api/sync and silently skips in cron.
  • Mutex only covers a single Node.js process. On Vercel, every function instance has its own lock — real protection comes from the VPS itself rejecting overlapping /connect calls, which means parallel syncs race and the loser gets a session with the wrong account.
  • Idempotency is strong: reruns of /api/sync converge rather than duplicate.

10 · Consolidated optimisation checklist

  1. Incremental history sync — stop fetching from 2000-01-01; use last_synced_at with a small overlap window for safety.
  2. Distributed VPS lock — replace sync-mutex.ts with a Postgres advisory lock (pg_try_advisory_lock) keyed on account login.
  3. Encrypt investor passwords at rest with pgcrypto + a key loaded from server env, so a leaked service-role key alone is insufficient.
  4. Materialised view for sessions (or incremental UPSERT touched-dates only) instead of full rebuild per sync.
  5. Denormalise user_id onto trades and sessions to simplify RLS and let Postgres use straight index scans.
  6. Queue-based cron — /api/cron/sync-all should enqueue per-account jobs (e.g. Supabase Queues, Upstash QStash, pg_cron) so long runs don't hit Vercel function timeouts.
  7. Partial-close handling in the normalizer — sum lot-weighted exit prices so reported pnl_pips matches reality for multi-leg exits.
  8. Cookie slimming — store only accountId in the mt5_account cookie; look up balance/equity on demand. Shrinks every request and avoids stale numbers.
  9. VPS health exposure — surface /health on the dashboard and block the Sync button if mt5_connected=false; today users can fire a sync that silently 500s.
  10. Observability — structured logs (request id, account id, duration, rows upserted) per sync run; the current codebase logs ad-hoc strings.
  11. cTrader adapter is a stub; the BrokerAdapter interface is in place, but wiring it will require a second VPS bridge or a shared process.
  12. Tests — VPS and mac normalizer have tests; API routes and the sync orchestration do not. Adding integration tests around /api/sync would catch most optimisation regressions.
